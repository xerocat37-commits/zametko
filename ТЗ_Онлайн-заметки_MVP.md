# Онлайн-заметки — Техническое задание (MVP)

> Единое рабочее пространство для учёбы. Локальное ПО для одного университета.
> Гибрид блокнота и доски объявлений для учебного процесса.

---

## 1. Роли

| Роль | Регистрация | Возможности |
|------|-------------|-------------|
| Студент | Сам: группа + ФИО + пароль | Личные заметки; чтение групповых и общих заметок |
| Препод | Сам: ФИО + пароль | То же + создание групп, приглашение студентов, групповые заметки |
| Заведующий (Head) | Создаётся админом, **не саморегистрация** | То же + общая рассылка (Global-заметки) |

На экране регистрации выбор только **«Студент» / «Препод»**. «Заведующий» — отдельный аккаунт (1–2 на учреждение).

---

## 2. Функциональность MVP

| # | Функция | Кто |
|---|---------|-----|
| 1 | Создание личных заметок | Все |
| 2 | Создание группы (название = специальность) | Препод |
| 3 | Приглашение студентов в группу | Препод |
| 4 | Создание групповых заметок | Препод |
| 5 | Чтение групповых заметок | Студенты группы |
| 6 | Общая рассылка / объявление | Заведующий |
| 7 | Просмотр рассылок | Все |

---

## 3. Архитектура

Функции 2–7 — общие для нескольких пользователей → нужен **сервер с общей БД**.
Чисто локальное приложение (БД только на устройстве) для этого **не подходит**.

```
[Клиент: .NET MAUI]  ──REST / HTTPS──>  [ASP.NET Core Web API]  ──>  [SQLite (WAL) на сервере]
```

- SQLite сохраняется — но на сервере. Обоснование из дипломной остаётся в силе.
- Клиент хранит только токен сессии и кэш; источник истины — сервер.

## 5. Основные сценарии

**Регистрация студента**
1. Выбирает «Студент» → вводит группу + ФИО + пароль.
2. Система ищет в GROUP_MEMBER запись `Invited` с этим ФИО и группой.
3. Совпадение → создаётся USER, проставляется StudentId, Status = `Active`.
4. Нет совпадения → «Вас не пригласили в эту группу».

**Регистрация препода** — выбирает «Препод» → ФИО + пароль → USER с Role = Teacher.

**Создание группы** — препод вводит название специальности → создаётся GROUP (TeacherId = свой).

**Приглашение студента** — препод в своей группе вводит ФИО → GROUP_MEMBER со Status = `Invited`.

**Заметки**
- Личная — NoteType = Personal, видна только автору.
- Групповая — препод создаёт, NoteType = Group, TargetGroupId = группа; видна членам группы.
- Рассылка — заведующий создаёт, NoteType = Global; видна всем.

## 6. Что исправить / уточнить

| # | Проблема | Решение |
|---|----------|---------|
| 1 | Противоречие: в тексте «только 2 роли», но в схеме и MVP есть «Заведующий» | Ролей **3**; саморегистрация — только для 2 |
| 2 | Нет пароля: вход по одному ФИО → любой входит под чужим именем | Добавлен `PasswordHash`; вход = ФИО (+группа) + пароль |
| 3 | Дипломная описывает прототип на **локальном** SQLite (MAUI), но группы/приглашения/рассылки требуют общих данных | Нужен сервер; SQLite переносится на сервер |
| 4 | В схеме негде хранить приглашение до регистрации студента | В GROUP_MEMBER добавлены `StudentFullName` + nullable `StudentId` + `Status` |
| 5 | Дублирование: членство в группе хранится и в USER.GroupName, и в GROUP_MEMBER | Источник истины — GROUP_MEMBER; GroupName в USER — только введённое значение |
| 6 | У GROUP_MEMBER не было первичного ключа | Добавлен `Id` (PK) |
| 7 | Введение дипломной обещает «файловое хранилище + мессенджер», которых нет в MVP | Перенести в раздел «будущие возможности» либо убрать из введения |
